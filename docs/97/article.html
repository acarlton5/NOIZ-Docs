1. What Is an Interactive Plugin?

An interactive plugin is any tool that responds to or triggers:

  • Viewer input (chat messages, clicks, reactions)
  • ⚡︎dB support actions (subs, cheers, quest completions)
  • Stream state (live/on break, featured, Partner status)
  • Inventory unlocks or milestone events

Plugins enhance the stream experience and can appear:

  • As overlays
  • In chat
  • Inside viewer sidebars
  • Tied to Quest UIs or boost moments

2. Typical Plugin Use Cases

  • ⚡︎dB-triggered reactions (e.g., chat rain, explosions)
  • Stream polls or real-time voting widgets
  • Milestone or Quest completion trackers
  • Timer-based minigames or randomizers
  • Viewer name highlight effects
  • Collaborative goal bars (e.g., “raise 5k ⚡︎dB”)

If it engages the audience, you can build it.

3. Plugin Development Flow

  1. Plan the interaction

    • What triggers it?
    • What does it display or do?
    • Does it require ⚡︎dB, chat, or viewer input?

  2. Use the Plugin SDK Template

    • Fork or download the boilerplate
    • Customize the manifest with trigger types, visuals, cooldowns
    • Hook into SDK events (e.g., onSub, onQuestComplete)

  3. Preview & Test

    • Use NOIZ’s local sandbox
    • Simulate events using mock viewer data
    • Optimize for both desktop and mobile viewers

  4. Submit for Approval

    • Add metadata and preview images
    • Set unlock type (free or ⚡︎dB unlock)
    • Submit via Developer → Submit Plugin

4. Essential SDK Hooks

HookDescription
onTipFires when a viewer sends ⚡︎dB directly or via Sub
onQuestCompleteFires when a viewer completes a stream-linked Quest
onKeywordFires when a chat message matches the plugin’s keyword trigger
triggerOverlay()Renders an overlay element (animation or HTML)
playSound()Plays a plugin-approved audio cue
setCooldown()Prevents repeated spam (per viewer or global)

Every plugin must declare all hooks it uses in its manifest.

5. Plugin Design Best Practices

✅ Do:

  • Minimize input latency
  • Support chat-safe animations
  • Provide mobile-friendly scaling
  • Offer streamer customization (color, text, placement)
  • Only trigger effects when initiated by a viewer or streamer

❌ Don’t:

  • Auto-play sound or video on load
  • Simulate events (e.g., fake “you got a sub!”)
  • Manipulate chat logs or impersonate mods/staff
  • Store or transmit viewer data outside NOIZ

6. Testing Tools

The Plugin Sandbox tool lets you:

  • Emulate chat triggers
  • Simulate ⚡︎dB actions
  • Test multiple viewer identities
  • Switch stream states (offline → live → featured)

If your plugin passes sandbox testing and visual review, it’s ready for submission.

7. Optimization & Performance

Keep assets:

  • Under 5MB for overlay bundles
  • Async-loaded where possible
  • Responsive from 720p to 4K viewports
  • Compliant with sandbox and security rules

⚠️ Memory-heavy or slow plugins may be rejected, throttled, or unlisted until optimized.

8. FAQ

Can I reuse Twitch or Kick plugin code? → Only if fully rewritten for NOIZ standards.

Can I build audio-only tools? → Yes — must include volume controls.

Can I partner with artists? → Absolutely — combining visuals + interactivity is encouraged.

Can I test with real ⚡︎dB? → Test mode uses simulated ⚡︎dB; real usage requires streamer installation.

Can my plugin store viewer data? → No — stateless design only, unless using platform-native inventories.
